# Claw Code 路线图

## 目标

让 Claw Code 成为最**易用且可靠**的 AI Agent 编程工具：
- 不依赖脆弱的提示词注入时机
- 会话状态透明可见
- 插件和 MCP 失败清晰可查
- 日常恢复无需人工干预

## 核心原则

1. **状态机优先** — 每个工作器都有明确的生命周期状态
2. **事件优先于日志** — 结构化事件优于文本日志
3. **先恢复再上报** — 已知失败模式先自动恢复再寻求帮助
4. **分支新鲜度优先** — 在将失败归咎于代码前先检测过时分支
5. **部分成功也是成功** — 结构化降级模式报告
6. **终端是传输层而非事实源** — 编排状态独立于终端
7. **策略可执行** — 合并、重试、变基、清理、上报规则应机器执行

## 当前痛点

### 1. 会话启动脆弱
- 信任提示会阻止 TUI 启动
- 提示可能出现在 shell 而不是 coding agent
- "会话存在"不代表"会话就绪"

### 2. 状态分散在多层
- tmux 状态
- 事件流
- git/worktree 状态
- 测试状态
- 插件/MCP 运行时状态

### 3. 事件太像日志
- 从嘈杂文本推断太多
- 重要状态未标准化为机器可读事件

### 4. 恢复循环太手动
- 重启工作器
- 接受信任提示
- 重新注入提示
- 检测过时分支
- 手动分类基础设施 vs 代码失败

## 路线图

### 第一阶段 — 可靠的工作器启动

#### 1. 工作器就绪握手生命周期
添加明确状态：
- `spawning` (启动中)
- `trust_required` (需要信任)
- `ready_for_prompt` (准备就绪)
- `prompt_accepted` (提示已接受)
- `running` (运行中)
- `blocked` (被阻塞)
- `finished` (已完成)
- `failed` (已失败)

验收标准：
- 在 `ready_for_prompt` 之前从不发送提示
- 信任提示状态可检测并发出事件
- shell 误投递成为可检测的一流失败状态

#### 2. 信任提示自动解决
为已知仓库/worktree添加自动信任白名单。

验收标准：
- 可信仓库自动清除信任提示
- 发出 `trust_required` 和 `trust_resolved` 事件
- 非白名单仓库保持门控

#### 3. 结构化会话控制 API
在 tmux 之上提供机器控制：
- 创建工作器
- 等待就绪
- 发送任务
- 获取状态
- 获取最后错误
- 重启工作器
- 终止工作器

验收标准：
- 可以不使用原始 send-keys 作为主控制平面

#### 4. 启动预检 / Doctor契约
在启动或提示工作器前，运行机器可读预检，报告通道是否真的安全。

预检应检查并发出类型化结果：
- 仓库/worktree 存在性和预期分支
- 相对于基础分支的新鲜度
- 信任门可能性 / 白名单状态
- 必需的二进制文件和控制套接字
- 插件发现 / 白名单 / 启动资格
- MCP 配置存在和服务器可达性预期
- 上次已知的失败启动原因（如有）

验收标准：
- 可以在启动注定失败的工作器前快速失败
- 阻塞启动返回简短结构化诊断而非强制窗格刮擦分类
- 可以总结"为什么这个通道甚至没有启动"而无需从终端噪音推断

### 第二阶段 — 事件原生集成

#### 5. 标准通道事件架构
定义类型化事件，例如：
- `lane.started` (通道已启动)
- `lane.ready` (通道就绪)
- `lane.prompt_misdelivery` (提示误投递)
- `lane.blocked` (通道被阻塞)
- `lane.red` (通道红色状态)
- `lane.green` (通道绿色状态)
- `lane.commit.created` (已创建提交)
- `lane.pr.opened` (已创建PR)
- `lane.merge.ready` (合并就绪)
- `lane.finished` (通道已完成)
- `lane.failed` (通道失败)
- `branch.stale_against_main` (相对于主分支过时)

验收标准：
- 消耗类型化通道事件
- Discord 摘要从结构化事件渲染，而不是仅从窗格刮擦

#### 6. 会话事件排序 + 最终状态协调
当同一会话在短时间内发出矛盾的生命周期事件时，必须暴露确定性最终事实，而不是让下游猜测。

所需行为：
- 为会话生命周期事件附加单调序列/因果顺序元数据
- 分类哪些事件是终结性的 vs 建议性的
- 将重复或乱序终结事件协调为一个规范通道结果
- 区分"会话终结状态未知因为传输失败"与真实"已完成"

验收标准：
- 可以在 `completed -> idle -> error -> completed` 噪音中存活而不会重复报告或信任错误的最终状态
- 会话事件爆发后的服务器中断表现为类型化不确定性状态，而不是静默重写历史
- 下游自动化每个通道/会话有一个规范终结结果

#### 7. 事件来源/环境标签
每个发出的事件都应说明它来自实时通道、合成测试、健康检查、重放还是系统传输层，以免将测试噪音误认为生产事实。

必需字段：
- 事件源类型 (`live_lane`、`test`、`healthcheck`、`replay`、`transport`)
- 环境/通道标签
- 发送者身份
- 下游自动化的置信度/信任级别

验收标准：
- 可以忽略或降低测试 ping 的排名，无需启发式文本匹配
- 合成/系统事件不会污染通道状态或触发虚假后续自动化
- 即使测试流量共享同一通道，事件流仍然保持机器可信

### 第三阶段 — 分支与集成质量

#### 8. 分支新鲜度门控
在运行可能只是错过主分支修复的大范围测试之前，强制显式分支新鲜度检查。

验收标准：
- 过时分支检测在启动前运行
- 明确选择加入可以跳过过时分支检查
- 过时分支状态发出明确事件

#### 9. 插件/MCP 启动健康报告
为每个插件/MCP 服务器暴露结构化启动结果，区分：
- 完全成功启动
- 部分成功启动（某些服务器工作，某些失败）
- 握手失败
- 配置错误
- 降级模式运行

验收标准：
- 无需从终端噪音推断插件/MCP 启动健康
- 部分成功启动是一流状态，而不是隐藏失败

### 第四阶段 — 自动化与编排

#### 10. Lane 状态持久化
在 `.claw/lane/` 目录中持久化结构化通道状态，这样人类或其他爪子可以从之前停止的地方继续，而无需解析聊天历史。

验收标准：
- 通道重启可以从持久化状态恢复
- 通道历史在会话间保持机器可读
- 人类可以检查 `lane.status` 而无需解析终端

#### 11. 自动重试策略
为已知的瞬时失败添加配置的自动重试：
- API 速率限制
- 暂时性网络问题
- 工具执行瞬时失败

验收标准：
- 无需人工干预重试已知瞬时失败
- 重试次数和退避策略可配置
- 重试尝试发出事件

### 第五阶段 — 开发者体验与文档

#### 12. 改善的开发体验
- 更好的错误消息，包含修复建议
- 更友好的入门引导
- 交互式教程
- 视频演示

#### 13. 完善文档
- 架构文档
- API 文档
- 插件开发指南
- 示例和用例
